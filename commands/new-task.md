# /llm-wiki:new-task

오늘 날짜의 작업 명세 파일을 생성한다. 기본은 직전 명세의 미완료 항목
(`[ ]` ready, `[WIP]` 사용자 작성 중) 을 제목+본문 통째로 이어받는다.
`--clean` 옵션을 주면 빈 템플릿으로 시작한다.

## 입력

- `$ARGUMENTS` 에 `--clean` 이 포함되면 carry-over 를 건너뛴다.

## 동작

1. **날짜 확인**: 환경 컨텍스트의 `currentDate` (YYYY-MM-DD) 를 `YYYYMMDD` 형식으로 변환 (예: 2026-05-21 → 20260521). 이후 "오늘 파일" = `docs/task/20260521.md`.
2. **오늘 파일 존재 확인**: `ls docs/task/<YYYYMMDD>.md` 또는 Read 로 확인. **존재하면** "이미 있습니다 — 경로" 만 출력하고 종료. 덮어쓰기 절대 금지.
3. **`--clean` 분기**:
   - `--clean` 이면 [Step 5] 의 "빈 템플릿" 분기로 직행.
   - 아니면 [Step 4] 의 carry-over 진행.
4. **이전 명세 carry-over** (기본):
   1. `ls docs/task/` → 파일명이 `YYYYMMDD.md` 패턴이고 오늘 이전인 것 중 **가장 최근** 파일을 선택 (예: `20260520.md` < `20260521.md`). 후보가 없으면 carry 없이 [Step 5] 빈 템플릿.
   2. 직전 명세를 Read.
   3. **TODO 리스트 파싱**: `## TODO` 섹션의 각 라인에서 마커 추출.
      - `N. [ ] <title>` → carry 대상 (ready)
      - `N. [WIP] <title>` → carry 대상 (user-writing)
      - `N. [DONE] <title> → <log path>` → 제외
   4. **본문 섹션 파싱**: 각 carry 대상 번호 `N` 에 대응하는 `## N.` 섹션 (다음 `## ` 또는 EOF 까지) 을 본문으로 함께 추출.
   5. **번호 재할당**: carry 대상을 등장 순서대로 1부터 재번호. TODO 라인과 본문 헤더 (`## <new_num>.`) 모두 갱신.
5. **파일 작성**:
   - `docs/task/_TEMPLATE.md` 를 Read 로 읽는다. 없으면 사용자에게 "`/llm-wiki:init` 을 먼저 실행하라" 안내하고 종료.
   - 헤더 `# YYYY-MM-DD` 를 오늘 ISO 날짜 (예: `# 2026-05-21`) 로 치환.
   - **carry 항목이 있으면**: 템플릿의 placeholder TODO 라인 (`1. [ ] <짧은 제목>`) 과 `## 1.` 빈 섹션을 제거하고, 그 자리에 재번호된 carry TODO 리스트 + 본문 섹션들을 삽입.
   - **carry 항목이 없거나 `--clean` 이면**: 템플릿 그대로.
   - `docs/task/<YYYYMMDD>.md` 로 Write.
6. **보고**: 생성된 경로를 출력하고, carry 가 발생했으면 다음을 함께 알린다.
   - 가져온 항목 수와 각 항목의 (새 번호, 원본 번호, 원본 마커, 제목)
   - 빈 템플릿이면 "사용자가 직접 TODO 항목과 본문을 채워야 한다" 안내.

## 예시

직전 `20260520.md` 가 다음과 같다고 하자.
```
## TODO
1. [DONE] A 작업 → docs/logs/20260520/a/
2. [ ] B 작업
3. [WIP] C 작업

## 1.
**목표:** A...

## 2.
**목표:** B...

## 3.
**목표:** C...
```

`/llm-wiki:new-task` 로 생성된 `20260521.md` 는 다음과 같다.
```
# 2026-05-21

## TODO
1. [ ] B 작업
2. [WIP] C 작업

## 1.
**목표:** B...

## 2.
**목표:** C...
```

`/llm-wiki:new-task --clean` 이면 템플릿 그대로 (TODO 1개, `## 1.` 빈 섹션).

## 제약

- 기존 파일 덮어쓰기 금지
- 사용자가 추가한 새 항목은 자동 생성 금지 — carry 만 수행
- `[WIP]` 항목은 사용자가 본문을 쓰는 중일 수 있으므로 본문도 그대로 복사 (가공·정리 금지)
- 사용자에게 "carry 한 항목이 무엇인지" 명시적으로 보고. 사용자가 원하면 직접 다시 정리할 수 있게.
- git add / commit 자동 수행 금지
